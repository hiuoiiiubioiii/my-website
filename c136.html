The free() Function in C

A continuous, in-depth breakdown of Dynamic Memory Deallocation

1. Introduction & Context

In C programming, we use Dynamic Memory Allocation (DMA) functions like malloc(), calloc(), and realloc() to reserve memory in the Heap at runtime.

The Problem: Local variables (which live on the Stack) are destroyed automatically when a function ends. However, dynamically allocated memory on the Heap is never released automatically. If you don't release it, it stays locked forever until the program terminates.

The Solution: We use the free() function to manually release dynamically allocated memory back to the Operating System.

2. Syntax & Basics

  • Purpose: To release/deallocate dynamically allocated memory.
  • Library: It is a built-in function defined inside <stdlib.h>.
  • Return Type: void (It does not return anything).
// Standard Syntax
void free(pointer);

// Example Usage
int *ptr = (int*)malloc(2 * sizeof(int));
// ... perform computations ...
free(ptr); // Releasing the memory

3. Visualizing the Heap: Before & After free()

To truly understand free(), we must look at what happens behind the scenes in your computer's memory. Let's assume we allocated space for two integers (10 and 2) using malloc, and the OS gave us memory starting at address 1000.

Phase 1: Before calling free(ptr)

The pointer ptr sits on the stack, holding the address 1000. It points directly to the active memory block in the Heap.

Stack

ptr = 1000
➔

Heap

Address: 1000
[ 10 | 2 ]

Phase 2: After calling free(ptr)

The memory block at address 1000 is released back to the Operating System. The OS can now give this space to another program or another malloc call.

Crucial Exam Concept: Does free() erase the data?

NO! The function free() does not overwrite, delete, or zero-out the data. The numbers 10 and 2 are technically still sitting in that physical memory location. You have simply told the OS: "I am done using this space; you can reuse it."

Stack

ptr = 1000
➔ ✂️

Heap

Address: 1000 (Released)
[ 10 | 2 ] (Data remains, but unsafe)

4. The Danger Zone: Dangling Pointers

Look closely at the "After Free" visualization above. Even though the Heap memory is released, the pointer `ptr` on the stack still holds the value `1000`.

What happens if you try to print it?

free(ptr);
printf("%d", *(ptr + 1)); // Trying to access the 2nd element

This results in Undefined Behavior (UB). Because the memory no longer belongs to you, the OS might have overwritten it with garbage values, or your program might crash entirely (Segmentation Fault). In the video demo, accessing freed memory resulted in unpredictable output like 64136, 131264, 3.

The Definition

A pointer that points to a memory location that has been deleted or freed is called a Dangling Pointer. It is "dangling" because the object it was pointing to is gone.

The Best Practice Fix

To defeat a dangling pointer, you must manually sever its connection to the old address immediately after freeing it.

free(ptr);
ptr = NULL; // The pointer is now safe and points to nothing.

5. The Double Free Error

What happens if you accidentally call free() twice on the same pointer?

free(ptr);
// ... some code ...
free(ptr); // ERROR!

This will cause a Core Dump (Double Free Error) and your program will instantly crash. The OS panics because you are trying to release memory that has already been marked as released.

Note: If you followed the Best Practice above and set ptr = NULL after the first free, the second free(NULL) will safely do nothing! This is why assigning NULL is so highly recommended.

6. Functions & Memory Leaks

The instructor demonstrated a specific scenario using VS Code where dynamic memory behaves very differently from static stack memory.

int* display() {
    // Allocating memory dynamically inside a function
    int *ptr1 = (int*)malloc(3 * sizeof(int));
    
    // ... assign values ...
    
    return ptr1;
}

int main() {
    int *main_ptr = display();
    
    // You MUST free it here!
    free(main_ptr);
    return 0;
}

The Lesson: If this were a standard local variable, it would die when display() ends. But because it was created with malloc, Heap memory survives function exits.

If you call a function that allocates dynamic memory and returns a pointer, it becomes your responsibility to free() that pointer in your main() function. If you forget, the memory stays locked while the program runs, causing a Memory Leak.

7. Summary Study Checklist 🎓

  • Always include <stdlib.h> when dealing with dynamic memory.
  • Stack memory frees itself automatically; Heap memory must be freed manually using free().
  • free(ptr) releases memory back to the OS; it does not erase the data inside it.
  • Accessing a freed pointer is Undefined Behavior.
  • A freed pointer immediately becomes a dangerous Dangling Pointer.
  • Golden Rule: Always write ptr = NULL; immediately after free(ptr); to prevent Double Free errors and dangling pointer bugs.
  • Always verify that memory was successfully allocated before using it (e.g., if (ptr == NULL) { /* handle error */ }).
