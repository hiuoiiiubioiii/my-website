Language Translators

Programming in C • In-depth Lecture Notes

1. Why do we need translators?

Computers process information in Binary Language (Machine Code), which consists entirely of 0s and 1s. However, as humans, we write programs in High-Level Languages (like C, Java) or Assembly Language, which use English-like words and syntax.


The Solution: A middleman software is required to translate our human-readable source code into computer-readable machine code. These are called Language Translators.

2. The Assembler

An Assembler is a highly specific translator designed for Assembly Language. In Assembly, programmers write code using "mnemonics" (e.g., ADD, SUB). The computer cannot understand mnemonics natively.

3. High-Level Translators: Compiler vs. Interpreter

To convert High-Level Languages (HLL) into machine code, we use either a Compiler or an Interpreter. While they serve the same ultimate purpose, how and when they perform the translation is entirely different.

Compiler

Execution Model: Takes the entire program at once. Translates everything into a permanent object file before execution begins.


Object Code: Generates a separate Object Code file (e.g., an .exe file). You can run this file multiple times without recompiling.


Execution Speed: Faster. Because the code is fully translated beforehand, the CPU just runs the machine code directly.


Debugging: Difficult. It scans the whole code and gives you a massive list of errors at the very end. You must fix all errors before an executable is created.


Memory: Requires more memory because it has to store the newly created Object Code file on your disk.


Source Code: Once compiled, you do not need the original source code to run the program. (e.g., You can share the .exe file).

Interpreter

Execution Model: Takes the program line-by-line. Translates one line and executes it immediately before moving to the next.


Object Code: Does not generate an intermediate Object Code file. Translation and execution happen directly.


Execution Speed: Slower. The CPU must pause execution to translate every single line, every time the program is run.


Debugging: Easy. If it finds an error on line 3, it stops execution immediately and points it out. You fix it, and it moves on.


Memory: Requires less memory as it translates on-the-fly without saving an intermediate file to the hard drive.


Source Code: You always need the original source code in memory every time you want to run the program.

4. The Real-Life Analogy (Visualized)

To truly understand the difference, consider the instructor's analogy of translating Russian to English:

🗣️ The Interpreter (Line-by-Line)

Imagine the Russian Prime Minister visiting your country. He speaks Russian, you speak English. A human translator stands between you. The PM speaks one sentence, stops, and the translator translates it. Then the next sentence. This is slow, continuous work.

🎬 The Compiler (Pre-translated)

Imagine watching a Russian movie with English subtitles. The entire movie was translated before you started watching. You don't have to pause the movie for someone to translate it for you. You just hit play and it runs fast and smooth.
