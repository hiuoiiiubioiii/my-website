Data Types in C

Integer Ranges, Format Specifiers, and the "Circle" of Overflow

1. The Core Problem

In standard C programming, you might declare a variable and assign it a specific number. But when you run the program, it prints a completely different number. Why?

The Answer: It has to do with the memory size of the data type, its mathematical range, and how the compiler wraps numbers around when they are too big or too small (overflow and underflow).

Note: In this lecture, we use the context of a 16-bit compiler where an integer takes 2 bytes. The logic applies perfectly to modern 32/64-bit systems (where int is 4 bytes), just with larger numbers.

2. Understanding Integer Storage

An int takes 2 Bytes of memory. Since 1 Byte = 8 bits, 2 Bytes = 16 bits. Computers only understand Binary (0s and 1s).

The Two Types of Integers

  • Unsigned int: 0 to 65535 (Stores strictly positive numbers).
  • Signed int (Default): -32768 to 32767 (Stores positive and negative numbers). We get this by dividing the total capacity (65536) by 2.

3. The "Circular" Concept (Crucial)

In C, when a number exceeds its maximum limit, the program does not crash. Instead, it wraps around like a clock. This is why format specifiers dictate what you see on the screen.

Unsigned Int %u

Starts at 0, goes clockwise up to 65535, then wraps back to 0.

Signed Int %d

Clockwise: Positive (0 to 32767). Counter-Clockwise: Negative (-1 to -32768).

4. Code Execution & Dry Runs

Let's look at what the compiler actually prints when we test these boundaries in C.

#include <stdio.h>
#include <conio.h>

void main() {
    int a = /* Value */;
    clrscr();
    printf("%d", a); // %d reads the Signed Circle
    printf("%u", a); // %u reads the Unsigned Circle
    getch();
}
1 Valid In-Range Number
int a = 32767;
a
32767
Address: 1000

What happens: 32767 is inside the valid Signed range. It sits happily in memory.

Output (%d): 32767 Output (%u): 32767
2 Exceeding the Positive Range (Overflow)
int a = 32768;
a
-32768
Memory strictly stores Bits

What happens: 32768 is too big! On the Signed Circle, stepping 1 past 32767 wraps you directly to the bottom left: -32768.

But if you ask %u to read it, it looks at the Unsigned Circle where that binary pattern equals 32768.

Output (%d): -32768 Output (%u): 32768
3 Going Further Out of Range
int a = 32770;

What happens: Start at 32767 on the Signed Circle. Take 3 steps forward (68, 69, 70). You wrap around to -32768 -> -32767 -> -32766.

Output (%d): -32766 Output (%u): 32770
4 Negative Value with Unsigned Format
int a = -10;

What happens: -10 is a valid signed integer. But %u refuses to print negatives. It looks at the Unsigned Circle, starts at 0, and steps backwards 10 spaces (0 -> 65535 -> 65534...).
Math: 65536 - 10 = 65526.

Output (%d): -10 Output (%u): 65526
5 Unsigned Int taking a Negative
unsigned int a = -32767;

What happens: %d sees the binary in memory and simply says "-32767". But %u forces the Unsigned Circle logic. Starting from 0 and going backward 32,767 steps lands you precisely halfway across the right side.
Math: 65536 - 32767 = 32769.

Output (%d): -32767 Output (%u): 32769
