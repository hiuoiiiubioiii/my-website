Memory Leak in C

A continuous, visual breakdown of how memory behaves, breaks, and is fixed.

1. The Core Concept

A Memory Leak is a heavily asked interview question for C and C++ developers. It occurs when a program allocates memory dynamically, but fails to release it back to the system after use.

Important Note: This problem does not exist in languages like Java or C#. Those languages feature an Automatic Garbage Collector—a background process that automatically deletes orphaned memory. In C/C++, the programmer has manual control (and manual responsibility).

2. Program Memory Layout

When the OS runs your program, it divides the memory into four specific sections. The core rule to understand is that the bottom three sections are fixed in size, while the Heap is variable.

Heap (Dynamic) ↕️ Grows upwards. Used by malloc/calloc. Not fixed!
Stack ↕️ Local variables & Function Calls. Fixed total size.
Global / Static Global variables outside functions.
Code / Text Compiled program instructions.

A memory leak is exclusively a problem caused by the improper use of the Heap section.

3. How the Stack Works (No Leak Here)

Let's look at standard local variables. When functions are called, they create a Stack Frame. When the function finishes, the Stack Frame is automatically destroyed. Memory is cleaned up perfectly.

int sum() {
    // Stored safely in the Stack Frame
    int a = 10, b = 11;
    int total = a + b;
    printf("%d", total);
} // <- Stack Frame destroyed here! Memory released.

int main() {
    int ch = 1;
    while(ch) {
        sum(); // Called repeatedly
        printf("Continue? (1/0): ");
        scanf("%d", &ch);
    }
}

Even though sum() is called over and over in a loop, there is no leak. The OS gives the memory to a and b, and then immediately takes it back when the function hits the closing bracket }.

4. Creating the Memory Leak

Now, we introduce dynamic memory allocation using malloc(). This allocates memory in the Heap, returning a pointer to the Stack.

int sum() {
    // ptr is in the Stack. The 8 bytes are in the Heap.
    int *ptr = (int*)malloc(2 * sizeof(int)); 
    
    // ... code using ptr ...
    
} // <- Stack frame destroyed. 'ptr' is erased.
  // BUT the 8 bytes in the Heap stay forever!

1. ptr is erased from the Stack automatically.
2. The 8 bytes in the Heap remain, but we lost their address.
3. Because this is in a while loop, every loop adds 8 *more* orphaned bytes. The heap explodes. Program crashes.

5. The Fix: free()

Because C does not have a Garbage Collector, you must manually release the Heap memory before the pointer in the stack gets destroyed. We do this using free().

int sum() {
    int *ptr = (int*)malloc(2 * sizeof(int)); 
    
    // ... use the memory ...
    
    free(ptr); // <- FIX: Memory returned to the OS!
}

Now, when the loop repeats, malloc will simply reuse the exact same 8 bytes. The Heap stops growing, and the program is stable.

6. Proof: Task Manager Demonstration

In the video, the instructor proves this by allocating a massive chunk of memory: malloc(40000 * sizeof(int)) and watching the Windows Task Manager.

Without free() ❌

  • Start: 0.4 MB RAM
  • Loop 1: 0.5 MB RAM
  • Loop 2: 0.6 MB RAM
  • Loop 3: 0.8 MB RAM

The memory steadily increases on every loop iteration until the system crashes.

With free() ✅

  • Start: 0.4 MB RAM
  • Loop 1: 0.4 MB RAM
  • Loop 2: 0.4 MB RAM
  • Loop 3: 0.4 MB RAM

The memory stays perfectly flat because the OS recycles the freed space.
